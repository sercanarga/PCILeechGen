# VFIO-user Firmware End-to-End Test Design

## Objective

Build a deterministic end-to-end test suite under `vfio-user/` that validates every supported donor fixture through the real PCILeechGen artifact-generation path and then exercises the resulting device identity and behavior from a Linux guest through VFIO-user and QEMU.

Vivado synthesis, implementation, bitstream generation, FPGA flashing, and physical-host validation are outside this scope.

## Device Matrix

The suite covers every fixture currently generated by `tests/cocotb/generate_all.sh`:

| Fixture | Guest acceptance criterion |
| --- | --- |
| `nvme` | PCI enumeration, NVMe driver bind, controller reaches live state, Identify succeeds |
| `sata` | PCI enumeration, AHCI driver bind, controller and implemented port initialize without fatal errors |
| `xhci` | PCI enumeration, xHCI driver bind, host controller initializes |
| `audio` | PCI enumeration and expected audio-class driver bind when the guest kernel provides it |
| `ethernet` | PCI enumeration, expected network driver bind, network interface creation |
| `wifi` | PCI enumeration and expected wireless driver bind when the guest kernel and required firmware provide it |
| `gpu` | PCI enumeration and expected display driver bind when the guest kernel provides it |
| `thunderbolt` | PCI enumeration and expected driver bind when the guest kernel provides it |
| `multibar` | PCI enumeration and all generated BARs expose the expected size and flags |
| `generic` | PCI enumeration and generated identity, capability chain, and BAR layout match the fixture |

Missing optional guest drivers or external device firmware produce an explicit skip only when the test declares that dependency. Enumeration, config-space, capability, and BAR checks remain mandatory for every fixture.

## Test Boundary

Each case follows one pipeline:

1. Run the repository's `pcileechgen build --from-json ... --skip-vivado` command for the fixture and selected board.
2. Validate that the required generated manifest, config-space image, BAR data, SystemVerilog, and TCL artifacts exist and agree on device identity and BAR layout.
3. Run the existing Cocotb/Verilator firmware-facing tests selected for that device class.
4. Start one generic VFIO-user server configured from the generated output.
5. Attach that server to QEMU and boot a deterministic minimal Linux guest.
6. Collect structured guest results for PCI discovery, config space, BARs, driver binding, kernel messages, and the device-specific smoke check.
7. Return a nonzero exit status for any mandatory failure and retain per-device logs.

The VFIO-user stage does not claim to execute the generated FPGA bitstream. It validates that the generated host-visible contract can be consumed by a real Linux PCI driver. Cocotb/Verilator remains responsible for exercising the generated HDL behavior.

## Architecture

### Orchestrator

A single strict-mode runner owns fixture selection, artifact generation, image setup, per-device timeouts, cleanup, and result aggregation. It accepts one fixture or the complete matrix. It never hides failures with unconditional `|| true` or pipeline truncation.

### Artifact Loader

The server reads generated identity and layout data rather than embedding vendor IDs, device IDs, class codes, capability offsets, BAR sizes, or BAR contents in C source. The loader rejects missing files, inconsistent sizes, overlapping regions, malformed capability chains, and unsupported BAR definitions before QEMU starts.

The generated build output is the source of truth. The fixture JSON is used to initiate generation and to label the test, not as a parallel runtime model.

### Generic PCI/VFIO Core

One focused core owns libvfio-user lifecycle, PCI config-space access, BAR region registration, reset, DMA-region callbacks, IRQ setup, and server shutdown. It has no device-class policy beyond selecting a behavior module through a small interface.

### Device Behavior Modules

Device-specific behavior is isolated behind a narrow read/write/reset interface. Modules implement only state that cannot be served directly from generated static BAR data and that is required for the corresponding guest acceptance criterion.

The first implementation must reuse or extract the existing NVMe behavior rather than keep `nvme_vfio_server.c` as a second server. AHCI, xHCI, and other class behavior must live in separate compilation units. Unsupported accesses return defined PCI/VFIO errors or deterministic reset values; they do not silently succeed.

### Guest Probe

The initramfs contains a small probe that emits stable key-value or JSON records. Host-side pass/fail logic consumes those records instead of grepping presentation-oriented prose. Raw `dmesg`, server logs, QEMU logs, and probe output are retained for diagnosis.

## Correctness Rules

- All ten fixtures must pass artifact generation and mandatory generic PCI checks.
- Device identity observed by the guest must match the generated config-space artifact.
- BAR count, type, width, prefetchability, and size observed by the guest must match generated metadata.
- The standard capability chain must be finite, aligned, non-overlapping, and reachable from the capabilities pointer.
- Extended capabilities must be finite, aligned, and non-overlapping.
- Device reset must restore the generated reset-visible state and clear runtime queue/interrupt state.
- A device-specific test passes only after the expected driver state and smoke operation complete; PCI enumeration alone is insufficient where a behavior module exists.
- Timeouts, guest crashes, QEMU exits, server exits, and malformed probe output are failures.
- Skips name the missing optional guest dependency and do not mask generic PCI failures.

## Failure Handling and Reports

Every device receives an isolated work directory containing generated artifacts, `server.log`, `qemu.log`, `guest-results.json`, and a concise result record. The matrix runner prints one summary table and exits nonzero if any mandatory case fails.

Processes are terminated through a single cleanup path. Socket paths and temporary directories are unique per case, allowing cases to run sequentially without stale-state collisions and later permitting bounded parallel execution without changing device code.

## Code Quality Constraints

- Do not duplicate generated device constants in the VFIO-user implementation.
- Keep orchestration, artifact parsing, VFIO plumbing, guest probing, and device behavior in separate units.
- Prefer data tables and explicit interfaces over repeated class-condition branches.
- Reject invalid input at module boundaries and propagate errors to the matrix result.
- Remove banner comments, narration, comments that restate the next statement, and stale comments from touched `vfio-user/` code.
- Retain comments only for non-obvious protocol constraints, external ABI requirements, or rationale that the code cannot express.
- Do not perform unrelated repository-wide comment cleanup or refactoring.

## Test Strategy

Implementation follows red-green-refactor cycles:

1. Unit-test artifact parsing and validation with generated and malformed fixtures.
2. Unit-test generic config-space and BAR access semantics without QEMU.
3. Unit-test device modules with driver-like register sequences and reset/error cases.
4. Add an integration test that starts the server and performs libvfio-user region accesses.
5. Add one QEMU guest test for generic enumeration, then specialize acceptance checks fixture by fixture.
6. Run existing Go and Cocotb suites to detect regressions.
7. Run the complete ten-device VFIO-user matrix and verify failure propagation by injecting one known-bad artifact.

## Deliverables

- A maintainable generic VFIO-user test harness under `vfio-user/`.
- Automated coverage for all ten donor fixtures.
- Structured per-device logs and a matrix summary.
- Removal or decomposition of the current hard-coded NVMe-only server.
- Documentation for local prerequisites and exact single-device/full-matrix commands.

## Non-Goals

- Vivado installation or invocation.
- Synthesis, place-and-route, `.bit`/`.bin` validation, or timing closure.
- FPGA flashing or physical-host enumeration.
- Full functional emulation of every operation supported by each real donor device.
- Replacing Cocotb/Verilator with VFIO-user.
